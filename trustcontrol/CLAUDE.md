### Как работать с Данилом

- Он часто присылает **скрин без текста** — это скрин ошибки или дашборда. Смотри на скрин, анализируй сам, не проси объяснений.
- Если прислал **скрин логов** — читай логи, находи ошибку, предлагай фикс.
- Если прислал **скрин дашборда** — смотри что не так (пустые данные, 0 в полях, неправильный текст), объясняй причину.
- Не задавай лишних вопросов — Данил устал от переписок. Действуй, потом уточняй если реально нужно.
- Всегда пушить в **main** ветку (его постоянная инструкция).
- Не добавляй фичи сверх задачи. Не рефакторь без просьбы.
- **ПРАВИЛО МИГРАЦИЙ — ОБЯЗАТЕЛЬНО:**
  Любое изменение полей в `backend/models/` требует ПАРНОЙ миграции.
  Порядок действий:
  1. Меняешь модель в `models/`
  2. Запускаешь `alembic revision --autogenerate -m "описание"` (в папке `trustcontrol/`)
  3. Открываешь сгенерированный файл, проверяешь `upgrade()` и `downgrade()`
  4. Показываешь Данилу 1-2 строки: какие колонки/индексы будут добавлены/изменены
  5. Ждёшь «ок» — только потом коммитишь **модель + миграцию вместе одним коммитом**
  Никогда не коммить изменение модели без парной миграции.
- **КОМАНДЫ ДЛЯ ТЕРМИНАЛА — ВСЕГДА ПО ОДНОЙ, КАЖДАЯ В ОТДЕЛЬНОМ блоке кода.**
  Данил работает в Windows cmd и копирует команды по очереди. Если дать несколько
  строк в одном блоке — он вставит их разом, cmd склеит в одну строку (`cd Xset Y=...`)
  и всё падает с «не удается найти указанный путь». Поэтому: одна команда = один
  ``` блок ```, пронумерованы (1, 2, 3), с «нажми Enter» между ними. Никогда не давай
  многострочные блоки команд для ручного запуска.

Каскад в `audio_analyzer.py` (`analyze_audio_with_fallback`, если ISSAI включён):
1. ISSAI один (бесплатно) → даёт казахский текст или пустоту.
2. ГЕЙТ СВЯЗНОСТИ `_triage_issai_text` (gpt-4o-mini, если ISSAI дал ≥3 слов):
   определяет {coherent, category}. КРИТИЧНО: русскую речь ISSAI (казахская модель)
   превращает в СВЯЗНО-выглядящую казахскую КАШУ из 20+ слов — нельзя доверять ей
   вслепую, иначе теряем русский разговор. Поэтому отдельный гейт проверяет именно
   СВЯЗНОСТЬ (НЕ через gpt_analyze — тот специально «вычитывает смысл из мусора»).
     • coherent + personal БЕЗ признака обслуживания → PERSONAL без OpenAI ✓
       (ЗАМОК №2 `_looks_like_service_interaction`: перед дропом проверяем платёж
        ИЛИ маркер сервиса — приветствие/заказ/прощание. Есть признак → НЕ дропаем,
        идём в OpenAI. Длину НЕ учитываем: личная болтовня бывает длинной.)
     • coherent + noise (нет plausible_conversation) → IGNORE без OpenAI ✓
     • coherent + business → ВСЁ РАВНО OpenAI (фрод-критично, могут быть рус.вставки)
     • НЕ coherent (каша = говорили по-русски) → ОБЯЗАТЕЛЬНО OpenAI (не дропаем!)
   Гейт консервативен: сомнение → coherent=false → идём в OpenAI (тратим деньги, но
   не теряем разговор — правильная асимметрия для фрод-детекции).
2б. РУССКИЙ ГЕЙТ `russian_stt` (бесплатный, self-hosted) — НЕ финальный транскрипт,

На фроде НЕ жалеем — `_escalate_borderline_fraud` (audio_analyzer) подключает САМЫЕ

  1. `gpt-4o-audio-preview` (`_judge_fraud_via_audio`) — СЛУШАЕТ саму запись: тон,

  2. ПОЛНЫЙ `gpt-4o` — второе мнение по тексту (острее на намёке «между нами»).
Вердикты ДОРОГИХ моделей ЗАМЕНЯЮТ пограничную догадку. Асимметрия фрода (пропустить
вора дороже ложной тревоги): fraud_confidence=max(умные), fraud_attempt=any(умные);
если ОБЕ сказали «чисто» — ложная тревога снимается. Текст НЕ меняем (уже полный).
<40 фрода нет, >=75 уже уверен (FRAUD_HARD) — не эскалируем. У чистых разговоров
confidence≈0 → эскалация редкая → дорогая audio-preview зовётся редко. gpt_analyze
имеет параметр `model`. Решение Данила: на кассовый диалог не жалеть (full transcribe),
на фрод не жалеть тем более (full audio-preview, которая слушает) — экономим ТОЛЬКО на
болтовне, которую бесплатно режут гейты ISSAI/русского ДО любого OpenAI.

ЧЁТКО, а система ставит IGNORE — который день. Разбор реальной записи (125с, тихая
-30 dBFS казахская речь) вскрыл баг в `worker/issai_worker.py`: пост-фильтр
галлюцинаций был СТРОЖЕ декодера. Декодер смягчён (`log_prob_threshold=-2.0`) чтобы
держать неуверенную казахскую речь, но фильтр сразу выкидывал всё с
`avg_logprob < -1.15`. Тихая речь на int8-CPU имеет именно такую низкую уверенность
→ распознавалась и тут же стиралась → пустой транскрипт → IGNORE.
ФИКС: из фильтра убран порог по logprob (низкая уверенность ≠ галлюцинация — её
судьбу решает декодер). Остались только бесспорные признаки галлюцинации:
`no_speech>0.9` ИЛИ `compression_ratio>2.8`, оба env-настраиваемы
(`ISSAI_HALLUC_NO_SPEECH` / `ISSAI_HALLUC_COMPRESSION`, видны в `/health`).
Каждый принятый сегмент теперь логируется с метриками.
⚠️ ТРЕБУЕТ ПЕРЕДЕПЛОЯ ВОРКЕРА (cron auto-update подхватит за ≤1ч, либо вручную).
Доп. фикс воркера (08.06): нормализация громкости по 99-перцентилю вместо
np.max — один стук аппарата (peak≈0.98) блокировал усиление тихой речи → VAD
принимал речь за тишину → 0 сегментов → IGNORE.
РЕАЛЬНЫЙ КОРЕНЬ (08.06, финал): диагностика прогоном файла через воркер показала
`segments:0, text:""` — ISSAI (казахская модель) просто не брал РУССКУЮ речь.

первичный STT на OpenAI (см. блок «STT цепочка» выше). Это разворачивает

## Суть проекта

SaaS-платформа для ИИ-мониторинга качества обслуживания.
Аналог системы "Петти" от Burger King — для малого бизнеса Казахстана.

**Как работает:**
1. Микрофон на кассе → PWA в браузере (`frontend/mic/index.html`)
2. Аудио отправляется POST-запросом на `/api/reports/submit` — **целый разговор одним файлом** (не кусками по 30с)
3. Сервер распознаёт речь: ISSAI → Yandex → OpenAI Whisper (фолбэк)
4. Анализирует тон и бизнес-события через `gpt-4o-mini-audio-preview`
5. Сохраняет в БД и отправляет отчёт в Telegram

## Ключевые архитектурные решения

### Единый движок оценки (analyzer.py → calculate_score) — ВАЖНО
ЕДИНСТВЕННЫЙ источник истины для балла 0-100. GPT даёт только СИГНАЛЫ
(events, tone, fraud_confidence, customer_satisfaction, energy_level) —
`calculate_score()` детерминированно считает из них финал. Один балл идёт
в `Report.score`, дашборд, аналитику сотрудников и тревоги.
Принципы (НЕ ломать — это ядро справедливости оценки):
- База 60. Приветствие/прощание/допродажа — БОНУС за наличие, НЕ штраф за
  отсутствие (микрофон обрезает начало через VAD, тихие визиты — норма).
- track_greeting/goodbye/upsell владельца уважаются всегда.
- Грубость штрафуется ОДИН раз (раньше был двойной штраф — баг).
- Фрод: порог уверенности. `>=75` (FRAUD_HARD) → балл=5 + критич. тревога;
  `50-74` (FRAUD_SOFT) → штраф −25, пометка fraud_status="suspected", БЕЗ тревоги.
- Контекст бизнеса (тип/скрипты/допродажи/оплата) идёт в промпт как ЛОГИКА,
  не как хардкод-примеры. Скрипт = смысловой ориентир, не дословный шаблон.
- `Report.gpt_score` — сырой сигнал GPT (для справки), `Report.score` — финал.
- Тихий визит (<6 слов / 1 спикер): сохраняется как нейтральный (is_short),
  выбрасывается только совсем пустой (<2 слов).
История: до мая 2026 было ТРИ несогласованных формулы (промпт GPT, calculate_score,
stats.py) + final_score никуда не сохранялся. Объединено в один движок.

### STT цепочка (audio_analyzer.py)
```
аудио → ISSAI (если ISSAI_WORKER_URL задан)
      → Yandex SpeechKit (если YANDEX_STT_API_KEY + FOLDER_ID заданы)
      → gpt-4o-mini-audio-preview (всегда, основной анализ тона)
      → Whisper-1 (fallback если аудио-модель не вернула транскрипт)
```
Yandex/ISSAI дают точные казахские СЛОВА → аудио-модель оценивает ТОН по звуку.

### PWA запись
- Весь разговор = один файл (не кусками по 30с)
- Авто-сброс через 180 сек (защита лимита 10MB)
- Формат WAV 16kHz моно

### Авторизация
- Владельцы → JWT токен
- Кассовые скрипты → `X-API-Key` заголовок

### Тарифы
```python
limits = {"trial": 1, "start": 1, "business": 3, "potok": 5, "network": 999}
```

### Смены (настраиваются владельцем)
- Дневная: владелец задаёт start/end в UTC+5
- Ночная: владелец задаёт start/end в UTC+5
- Kazakhstan timezone = UTC+5

### energy_level (1-5)
Вовлечённость кассира: 5=живой энергичный, 1=мёртвый роботичный.
Поле в `Report.energy_level`, отображается в аналитике сотрудников.
